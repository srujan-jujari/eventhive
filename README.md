# EventHive - Automated Event Registration & Ticketing System

EventHive is a modern, responsive, full-stack multi-page web application designed for streamlined event registration and ticketing. This repository contains the complete end-to-end DevOps pipeline for the application, fulfilling requirements for infrastructure provisioning, containerization, orchestration, testing, and CI/CD automation.

## Project Overview

The primary goal of this project is to demonstrate a real-world DevOps lifecycle. We transitioned a local Flask-based web application into a fully deployable, cloud-native project automated with modern CI/CD tools. 

## Tools Used
- **Application**: Python, Flask, Jinja2, SQLite, Bootstrap 5
- **Containerization**: Docker
- **Testing**: Pytest, Apache JMeter
- **Infrastructure as Code (IaC)**: Terraform (AWS EC2)
- **Configuration Management**: Ansible 
- **Orchestration**: Kubernetes (`kubeadm`, Flannel CNI)
- **CI/CD**: Jenkins

## Setup Instructions

### 1. Run Locally (Development)
1. Install Python 3.9+.
2. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
3. Run the application:
   ```bash
   python app.py
   ```
4. Access at `http://localhost:5000`. Log in with `admin:admin` to create events.

### 2. Run via Docker
1. Build the image:
   ```bash
   docker build -t eventhive:latest .
   ```
2. Run the container:
   ```bash
   docker run -p 5000:5000 eventhive:latest
   ```

## Cloud Deployment Steps

### Phase 1: Infrastructure Provisioning (Terraform)
Navigate to the `terraform/` directory to create the AWS EC2 server.
```bash
cd terraform
terraform init
terraform apply
```
This provisions a `t2.medium` instance and configures the Security Group (SSH, HTTP, Port 5000, NodePorts 30000+). Note the IP address output.

### Phase 2: Configuration (Ansible)
1. Add the Terraform output IP to `ansible/inventory.ini`.
2. Run the playbook to install Docker, set up Kubernetes, install Flannel networking, and prepare the node:
```bash
cd ../ansible
ansible-playbook -i inventory.ini playbook.yml
```

### Phase 3: CI/CD & Orchestration (Jenkins & Kubernetes)
1. Configure Jenkins to point to your repository.
2. The `Jenkinsfile` will automatically execute the following stages:
   - **Checkout**: Pulls the latest code.
   - **Build**: Creates the Docker image.
   - **Test**: Runs `pytest` inside the Docker image to verify core logic.
   - **Push**: Authenticates and pushes to DockerHub.
   - **Deploy**: Runs `kubectl apply` using the manifests in `k8s/` to deploy the app to your Kubernetes cluster and expose it via NodePort `30000`.

## Load Testing with Apache JMeter

To verify the app can handle multiple users booking tickets concurrently:
1. Open **Apache JMeter**.
2. Create a **Thread Group** (e.g., 50 Threads/Users, 5 seconds Ramp-up).
3. Add an **HTTP Cookie Manager** to your Thread Group (Flask relies on cookies for sessions).
4. Add an **HTTP Request** Defaults config element pointing to your Server IP and Port `30000` (if deployed via K8s NodePort) or `5000` (Local).
5. **Simulate Login**: 
   - Add an HTTP Request (POST to `/login`) with parameters `username=admin` and `password=admin`.
6. **Simulate Booking**:
   - Add an HTTP Request (POST to `/book/1`) – Assuming event ID 1 exists.
7. Add a **View Results Tree** listener and run the test. Verify there are no failures and tickets successfully decrement without race condition errors.
